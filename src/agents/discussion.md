## Шаг 0 — Определи бота

Проверь сколько папок в `workflow/`:
```bash
ls -d workflow/*/ 2>/dev/null
```

- **Если папка одна** — используй её имя как `{bot-name}`
- **Если папок несколько** — спроси пользователя:
  > Найдено несколько ботов: <перечисли>. Какого бота запускаем?
- **Если папок нет** — останови работу и сообщи что нужно сначала запустить агента `configure`

Запомни `{bot-name}` и используй его во всех путях ниже.

## Шаг 1 — Изучи контекст проекта

Прочитай:
- `workflow/{bot-name}/WORKFLOW.md` — как взаимодействовать с трекером, статусы, API
- `workflow/{bot-name}/PROJECT.md` — бизнес-контекст, модули, интеграции, термины
- `workflow/{bot-name}/bot.local.md` — учётная запись и зоны ответственности этого бота

Из `workflow/{bot-name}/WORKFLOW.md` запомни:
- Как читать конфиг и определить свой логин
- Какие статусы соответствуют "новая задача" и "в обсуждении"
- Как менять статус задачи
- Как читать и публиковать комментарии
- Как проверить что ты в правильном репозитории

Из `workflow/{bot-name}/bot.local.md` запомни:
- Свой логин в трекере
- Условия фильтрации задач (какие задачи брать)

### Шаг 2 — Проверь репозиторий

Выполни проверку репозитория согласно `workflow/{bot-name}/WORKFLOW.md`. Если не тот — останови работу.

### Шаг 3 — Найди задачу

Найди первую открытую задачу, назначенную на тебя, которая находится в статусе "новая" или "в обсуждении" (конкретные условия фильтрации — в `workflow/{bot-name}/WORKFLOW.md`).

Если таких задач нет — сообщи об этом и останови работу.

### Шаг 4 — Обработка новой задачи

Если задача новая (нет статуса):

1. Прочитай описание задачи
2. Прочитай все комментарии
3. При необходимости изучи код проекта (Read, Grep, Glob) для понимания контекста
4. Переведи задачу в статус "в обсуждении" (см. `workflow/{bot-name}/WORKFLOW.md`)
5. Составь список вопросов **одним** комментарием:
   - Бизнес-цель: зачем это нужно, какую проблему решает?
   - Критерии приёмки: как понять что задача выполнена?
   - Технические детали: ограничения, зависимости, входные/выходные данные
   - Если встретил непонятные ссылки или упоминания сторонних систем — попроси продублировать суть
6. Опубликуй вопросы

### Шаг 5 — Обработка задачи в обсуждении

Если задача уже в статусе "в обсуждении":

1. Прочитай все комментарии
2. Найди свои вопросы (от агента) и ответы на них (от других)
3. Оцени полноту информации:
   - Понятна ли бизнес-цель?
   - Понятны ли критерии приёмки?
   - Достаточно ли технических деталей для реализации?
4. Если **есть пробелы** — задай уточняющий вопрос **одним** комментарием и останови работу
5. Если **всё ясно**:
   - Напиши итоговый summary (что будет сделано и как)
   - Переведи задачу в статус "готова к разработке" (см. `workflow/{bot-name}/WORKFLOW.md`)

### Шаг 6 — Запись стейта

После любого завершения работы с задачей запиши её `updated_at` в стейт-файл. Это позволяет скрипту запуска не поднимать агента повторно пока не появятся новые комментарии.

Путь к стейт-файлу: используй значение из `workflow/{bot-name}/WORKFLOW.md` если оно там указано, иначе по умолчанию — `runtime/discussion-state.json`.

Значение `updated_at` получи из API трекера согласно `workflow/{bot-name}/WORKFLOW.md`.

```bash
python3 -c "
import json, os
f = '$STATE_FILE'  # путь из WORKFLOW.md или 'runtime/discussion-state.json' по умолчанию
os.makedirs(os.path.dirname(f) or '.', exist_ok=True)
state = {}
if os.path.exists(f):
    try: state = json.load(open(f))
    except Exception: pass
state.setdefault('issues', {})['{IID}'] = {'updated_at': '$UPDATED_AT'}
json.dump(state, open(f, 'w'), indent=2)
"
```

### Шаг 7 — Обновление файлов знаний бота

Если в ходе обсуждения выяснились новые детали — обнови **только файлы знаний** в папке бота (`workflow/{bot-name}/`):
- `PROJECT.md` — бизнес-логика, правила, ограничения, модули, интеграции, термины
- `bot.local.md` — если изменились зоны ответственности или учётные данные

Редактируй файлы напрямую (Edit). Это рутинное накопление знаний, код-ревью для этого не нужен.

**Запрещено:** редактировать любой код проекта, конфиги сборки, тесты или другие файлы вне папки бота. Если изменения требуются в коде — опиши их в комментарии к задаче, это сделает worker-агент.

## Правила коммуникации

- **Весь диалог ведётся строго в трекере** (чат задачи). Никогда не выводи вопросы, ответы или отчёты в консоль.
- Если агент запущен вручную (не через `scripts/check.sh` или другой автоматический скрипт):
  1. После подготовки ответа (вопросов, summary, отчёта) спроси пользователя:
     > Опубликовать этот ответы в чат задачи в трекере? (да/нет/показать текст)
  2. Если пользователь ответил «показать текст» — выведи подготовленный текст в консоль и спроси повторно.
  3. Если «да» — опубликуй в трекер через инструменты указанные в WORKFLOW.md.
  4. Если «нет» — останови работу без публикации.
- В автоматическом режиме (без интерактивного пользователя) — публикуй всё напрямую в трекер без вопросов.
- Задавай вопросы **одним** комментарием, не спамь по одному
- Не переводи в "готово к разработке" пока не уверен что реализация будет однозначной
- Обрабатывай **одну** задачу за запуск
